Design your first durable application.

A conceptual walkthrough for moving the important facts of an AI-assisted workflow out of ephemeral conversation and into an explicit lifecycle.

In this tutorial

Tutorial overview First application StateSpec basics Release status

No installation implied This is a design exercise using a hypothetical study planner. It is not a promise that a corresponding StatePort package or public template is available today.

The scenario

Imagine a study planner that helps someone prepare for an exam. A chat assistant could suggest a plan, but the learner still needs to see the current plan, upcoming work, decisions, and completed evidence after the chat ends.

The goal is not to make the assistant cleverer. The goal is to make the application’s truth legible and durable.

1. Separate facts from conversation

Write down the facts that must survive a new model, a compacted context window, or an offline review:

  • the learner’s goal and target date;
  • the current plan and its owner-approved changes;
  • evidence that a task was completed or checked;
  • the policy that determines who may make which changes.

A conversation can discuss these facts. It should not be the only copy of them.

2. Give every file an owner

Before allowing a runner to edit workflow files, record whether each path belongs to the template, the individual instance, generated output, or an explicit override. This turns “the agent changed something” into a reviewable question: was that agent allowed to change this exact material?

3. Model a small lifecycle

For one proposed plan adjustment, the lifecycle could be as simple as:

proposal ↓ reviewed authority ↓ applied change ↓ validation result ↓ receipt and current state

Each step has a different meaning. Keeping them distinct makes failures visible: an applied change can fail validation, and a locally validated change can still await human acceptance.

4. Make the assistant an adapter

The assistant may propose a plan, explain a state transition, or help fill a draft. It does not receive permission merely because it generated the idea. A host must declare what it can actually do, and any missing capability should degrade explicitly rather than silently changing the workflow.

5. Close with evidence

For the learner, a useful result is not just “done.” It is a clear current state plus enough evidence to understand what changed, what checked it, and what remains open. That same pattern helps a teacher, operator, or future maintainer reason about the application without replaying a conversation.

Takeaway

StateSpec starts with the durable boundary: facts, ownership, authority, lifecycle, and evidence. The AI layer becomes more useful when it operates within that boundary instead of replacing it.
