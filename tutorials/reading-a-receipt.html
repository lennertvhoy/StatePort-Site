Read a receipt without mistaking it for a verdict.

A short conceptual exercise for distinguishing what was proposed, authorized, applied, validated, and still awaiting a person’s judgement.

Design exercise The receipt below is a learning aid. Its example format is deliberately small and illustrative.

The situation

Imagine that an application proposes to move one study session from Wednesday to Thursday. A useful record should help the owner answer more than “did the assistant say it was done?”

Read the stages in order

proposal: move session / Wed → Thu proposal identity: sha256:… exact reviewable content authority: owner decision bound to that identity execution: one bounded local transition attempted apply result: applied validation: schedule rules passed receipt state: recorded human acceptance: not implied

The proposal identity matters because an approval for one plan should not drift into approval for a changed plan. The execution and apply result matter because a proposal can be authorized without running, and a run can fail before publication.

Ask five questions

  1. Which exact application instance and source identity does this concern?
  2. What content was prepared for review?
  3. What authority covered the act, and did it match the prepared content?
  4. What was actually applied and what validation was recorded?
  5. Which part still needs external acknowledgement or human acceptance?

Notice what a receipt cannot say

A receipt cannot prove that a learner’s schedule is wise, that a person understood the proposal, or that an external calendar update can be undone. Those questions need explicit policy, human judgement, and—when applicable—separate evidence from the external system.
