Early direction Agent Kits are a design proposal. Availability lives in the release ledger.

A useful shape for a focused tool

A study review guide, project kick-off assistant, or checklist steward should arrive with its shape intact: what it does, what it needs, and who owns the resulting work. The proposal is a template-first Agent Kit with a public template, runtime profile, policy request, and integrity metadata.

01

Public template

The workflow and declared views begin with an immutable, public source.

02

Declared runtime

A host adapter names its capabilities, sandbox, and limits.

03

Explicit authority

Effective capability remains the intersection of template request, instance grant, and operator policy.

04

Owned instance

A user starts fresh or deliberately imports a separately governed backup.

Docker-first, later

The first distribution profile under consideration is Docker/OCI. An image is a replaceable execution artifact; durable StateSpec data and credentials stay outside it. A kit should remain inspectable, mount an owned instance deliberately, and use the same lease, validation, transaction, and receipt contracts as agent-native use.

Separate from StateBench

A kit earns its value by solving a real use case. StateBench may later evaluate an exact configuration under controlled conditions; release trust and permissions remain separate concerns.

Before it ships

  • Publish an exact release source and its acceptance criteria.
  • Design a versioned schema, threat model, and public/private fixture policy.
  • Define ownership-aware install, update, export, import, rollback, and revocation behavior.
  • Bind Docker/OCI provenance and a release identity while keeping data and credentials outside image layers.
  • Validate one exact release candidate through CLI, application, degraded-host, and human-acceptance journeys.
