Capability-based qualification

StatePort separates the browser and durable-storage path from host-integrated execution. An environment is not supported just because a container starts: each capability needs an explicit demonstration, and the release position records how far that demonstration has gone.

Platform qualification matrix
Environment Browser and durable storage Host-integrated execution Release position
Linux Primary qualification target. Codex CLI and Linux isolation can be qualified. First supported release.
Docker Desktop for Linux Clean-install acceptance target. Requires explicit provider-authentication and sandbox qualification. Must pass before release.
macOS + Docker Desktop Expected UI and durable-storage path. Bubblewrap is Linux-specific; host adapters require separate proof. Preview only after testing.
Windows + Docker Desktop / WSL 2 Expected UI and durable-storage path. Filesystem, terminal, provider, and WSL behaviour require separate proof. Preview only after testing.

The matrix names a release contract, not a promise that every row is currently available. The host-portability page explains why provider identity alone cannot substitute for capability and degradation evidence.

The first clean-install user story

This is the first end-to-end scenario a later signed release must demonstrate. It is not an installation guide yet: the exact commands, image digests, and release identity belong with that release.

  1. Install the supported Docker and Compose environment.
  2. Start an exact signed StatePort release from an empty data directory.
  3. Complete first-run diagnostics.
  4. Import existing StateSpec repositories through an explicit, reviewed path.
  5. Instantiate a new application from an accepted canonical template.
  6. Configure Codex or another supported host without copying credentials into StatePort.
  7. Prove conversation, terminal, backup, restore, and governed-change workflows.
  8. Stop the old service only after data and recovery acceptance pass.

What does not qualify a platform

A successful image build, a container that answers one health endpoint, or a browser that reaches a landing screen is insufficient. Qualification needs the actual browser path, persistent state, provider route, host boundaries, governed change, backup, isolated restore, and failure behaviour to remain legible together.

Evidence before a support claim

Before this page can describe a platform as supported, the exact release and environment need a recorded acceptance run. The record should identify the source, images, host and adapter, authentication route class, storage location, checks performed, observed limits, and recovery outcome. A later release guide can then bind those facts to real artefacts rather than a moving branch.
