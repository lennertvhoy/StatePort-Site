Reading guide This page explores the model. Availability lives in the release ledger.

Install from an identity, not a folder name

Installability begins with a source that can be identified and checked. A release is a pinned identity, not a moving branch, matching directory name, or local fixture. Development candidates still have a useful role in isolated tests.

Use and understand

Application-first use means a person starts with their application’s conversation, actions, progress, and state. Provenance, ownership, capability state, approvals, and receipts are progressively disclosed when they are needed to understand what happened.

Update and move

A trustworthy update needs an exact source target, ownership-aware changes, conflict classification, a reviewable plan, validation, and a way to preserve the original instance if publication fails. Moving an instance has the same care: it is a migration with its own evidence.

Back up and recover

A backup is a bounded archive with an identity and validation record. Recovery needs its own demonstration: a dry run, isolated target, validation, cleanup, and proof that the live instance stayed intact. The release ledger records when lifecycle steps become available.

A useful reading rule

designed ≠ implemented implemented ≠ locally validated locally validated ≠ independently accepted accepted ≠ released

The release ledger is intentionally separate from conceptual documentation so a future installation guide can bind to a real version, source identity, and verification record.
