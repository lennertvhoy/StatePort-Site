Reading guide This page explores the model. Availability lives in the release ledger.

Application and instance

An application is a reusable definition: it describes intended state, requested capabilities, views, lifecycle rules, and validation expectations. An instance is the particular durable copy a person, team, or class uses. It holds its own identity, private state, grants, and history.

This distinction prevents a reusable template from quietly becoming the owner of a person’s data. A template can request a capability; it cannot grant itself that capability.

Canonical state and projections

Canonical state is the declared source of truth for a concern. A screen, summary, cached index, generated context pack, host session, or model response may be useful, but it must not silently outrank the source it represents.

template content → designated source private instance facts → instance state generated views → rebuildable projection provider session → noncanonical operational context external system facts → that external system

Capabilities are intersections

Effective capability is deliberately narrower than a request. It is the intersection of what the application requests, what the instance grants, what operator policy permits, what the current actor may do, and what the runtime can actually support. If one part is missing, the user should see an honest limit.

Approvals, grants, and receipts

A standing grant defines a bounded class of authority. A decision can authorize a particular consequential act. A receipt is a structured, checkable claim about what was prepared, authorized, applied, observed, and validated. It is stronger when it binds exact identities and evidence; it is never magic proof on its own.

Ownership categories

Before an automated runner changes a workflow file, its authority must be legible: template-owned, instance-owned, generated, or an explicit override. This turns “the agent changed something” into a checkable question about a specific path and rule.
