Replaceable execution needs declared limits.

A portable application does not require every host to behave the same. It requires the application’s durable boundary to survive while hosts declare what they can actually do.

Reading guide This page explores the model. Availability lives in the release ledger.

Host sessions are not the application

A model session, prompt, authentication observation, host configuration, or context window can be useful operational data. It must not override the instance’s policy, approvals, lifecycle truth, or durable evidence. The same application should remain legible through its repository and command-line tools even when a particular host is unavailable.

Capability before branding

Instead of assuming that a named host is interchangeable with another, a host should state its capabilities: structured events, cancellation, sandbox support, tool boundaries, output reporting, token telemetry, and the authentication route class. Missing support should produce an honest degradation, not an invented equivalence.

Where the host work stands

The public documentation describes a host-neutral model. A narrow direct-Codex conversation path has been exercised locally; Pi remains a reference-host direction. The release ledger will carry supported installation and host coverage as they are verified.

What stronger evidence would look like

Useful host evidence binds the exact application state, adapter and host versions, declared sandbox, model, context policy, authentication route class, validation commands, output artifacts, and observed failure/degradation modes. A provider label alone does not explain the result.
