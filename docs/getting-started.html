Design exercise This small repository shape is a way to think through an application. Availability lives in the release ledger.

1. Decide what must not be lost

Write down the things people will need later: goals, decisions, rules, who owns the work, what is happening now, and how you know it is right.

Keep the question narrow. A small set of clear facts is more useful than a grand file whose meaning depends on a particular agent’s prompt.

2. Make ownership clear

Separate the starting instructions, the project’s own files, generated work, and deliberate edits. Then it is clear what a tool may change and what it must leave alone.

example-application/ ├── AGENTS.md # operating constraints for people and agents ├── PROJECT_STATE.yaml # current, structured truth ├── PROJECT_DNA.yaml # durable product decisions ├── NEXT_ACTIONS.md # deliberately short active queue ├── BACKLOG.md # deferred work ├── WORKLOG.md # history, not current truth └── app/ # application-specific material

Files like these make the important facts easy to find without relying on a chat transcript.

3. Ask for access; do not hand out unlimited access

A tool can say what it needs to do. People and the project decide what it may actually do. “The workflow asked for it” should never be the whole permission check.

4. Keep the steps separate

Planning a change, approving it, doing it, checking it, and accepting it are different moments. Keeping them separate makes it easier to see what really happened.

5. Do not tie the work to one tool

The project should remain understandable if a model, app, or computer changes. Tools should say what they can do, and be honest about what they cannot.
