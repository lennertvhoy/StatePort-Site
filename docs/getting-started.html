Begin with a boundary you can inspect.

The first useful question is not “which model?” It is “what must remain true and understandable when the model is gone?”

Design exercise This small repository shape is a way to think through an application. Availability lives in the release ledger.

1. Name the durable facts

Start by identifying what someone should still be able to inspect tomorrow: goals, decisions, policy, ownership, current work, and evidence. These are application facts, not hidden model memory.

Keep the question narrow. A small set of clear facts is more useful than a grand file whose meaning depends on a particular agent’s prompt.

2. Make ownership visible

StateSpec separates template-owned material, instance-owned material, generated output, and explicit overrides. That distinction makes it possible to understand what a runner may update and what it must preserve.

example-application/ ├── AGENTS.md # operating constraints for people and agents ├── PROJECT_STATE.yaml # current, structured truth ├── PROJECT_DNA.yaml # durable product decisions ├── NEXT_ACTIONS.md # deliberately short active queue ├── BACKLOG.md # deferred work ├── WORKLOG.md # history, not current truth └── app/ # application-specific material

These familiar-looking files are useful because they make the application’s operating facts legible without relying on a chat transcript.

3. Ask for capabilities; do not grant them

A template can declare the capabilities it needs. The effective capability must still be the intersection of the request, the specific instance grant, and the operator’s policy. This keeps “the workflow asked for it” from becoming an authorization decision.

4. Preserve the difference between stages

Proposal, approval, execution, validation, and acceptance are different facts. A useful lifecycle does not collapse them into one optimistic status. Receipts help record what changed, under what authority, and what was actually checked.

5. Treat execution as replaceable

StateSpec is intended to be portable. A particular host can be a reference implementation, but an application should be able to declare its actual needs and receive explicit degradation when another host cannot provide the same capability.
