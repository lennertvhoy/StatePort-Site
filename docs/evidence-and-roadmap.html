Reading guide This page explores the model. Availability lives in the release ledger.

Four useful evidence states

implemented → locally validated → independently accepted → human accepted → released

These states do not collapse into one another. Local validation can be valuable and still be self-adjudicated. Independent CI can test an exact source without proving user value. Human acceptance can evaluate a product journey without converting an unresolved source into a release.

What is public today

This website brings together the documentation, prototype, paper, and release ledger. Software availability is recorded separately, with its exact source and evidence.

What remains open

  • A public source release, licensing decision, artifacts, and verified install path.
  • Independent validation of the exact delivered source and its packaged runtime.
  • Canonical upstream release acceptance for application packages.
  • External-user and human product acceptance.
  • Controlled evaluation before any comparative or performance claim.

How to evaluate the model

The central questions are testable: does explicit state improve recovery after a session loss? Do digest-bound approvals reduce mismatch? Do receipts improve a person’s understanding of change? Does host replacement preserve more useful continuity than a transcript export? A credible answer needs baselines, repeated runs, and measures that can falsify the claim.
