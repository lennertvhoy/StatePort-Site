Boundaries reduce risk. They do not make risk disappear.

StatePort’s model uses explicit authority, source identity, and evidence to make sensitive work more inspectable—not to claim certification or perfect safety.

Reading guide This page explores the model. Availability lives in the release ledger.

Threats worth naming

An application that uses a model can face malicious source material, prompt injection, poisoned state, stale approvals, confused authority, compromised hosts, forged evidence, unsafe external effects, and backup leakage. Treating an “agent” as one trust boundary makes these problems harder to see.

Design responses

  • Record source identity and reject untrusted promotion from a fixture or development candidate.
  • Give every workflow path an ownership rule before automated mutation.
  • Bind consequential approvals to exact proposals and preconditions.
  • Constrain execution hosts and make their capabilities and degradations explicit.
  • Keep secrets outside canonical state and avoid copying provider credentials into templates, images, or public material.
  • Use receipts and validation results to make changes reviewable after the fact.

Privacy is a design responsibility

Durable state and model context are different concerns. An application should make context-loading policy explicit, bounded, and measurable. Private notes, conversations, identifiers, credentials, and raw paths should not become fleet telemetry merely because they are convenient to aggregate.

What this does not establish

This documentation does not claim GDPR compliance, NIS2 compliance, certification, a penetration test, a complete threat model, or legal advice. Controls must be evaluated in the actual deployment, with real data flows, providers, retention, and operators in scope.
