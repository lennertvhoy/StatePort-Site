Durable state · governed change · replaceable execution

The idea

The application should hold its shape—even when the tools change.

Stateware gives an application a durable, inspectable core. Tools can work around it without becoming the only memory.

For StatePort, the visible product is the installed application. The operational machinery stays in the background until a person needs it.

01

Canonical source

Human-readable files remain the durable source of application truth—inspectable, repairable, and usable without one particular provider.

02

Governed change

Permissions, approvals, validation, and receipts are separate facts. A change is not simply “done” because a model produced text.

03

Replaceable execution

A workflow can declare its requirements while execution hosts and models declare their real capabilities and limitations.

A practical route

Build an application from durable decisions outward.

01 / DEFINE

Describe the application

Use a trusted StateSpec template to describe the state, capabilities, and views an application needs—without giving the template unlimited authority.

02 / OWN

Materialise an instance

Create a user-owned instance whose durable records can be understood outside the runtime that happens to be helping today.

03 / WORK

Use the application first

Conversation, actions, progress, and notifications belong to the application. Advanced platform controls appear only when they are useful and allowed.

04 / VERIFY

Keep evidence with change

Validation and receipts preserve the difference between a proposal, an applied result, and a result that has actually been checked.
