What must exist before this becomes a release

A release earns an “available” status only when its claims bind to exact, inspectable material.

  1. Public boundaryA clearly identified implementation source and product license decision.
  2. Versioned materialAn exact version, release notes, and immutable source reference.
  3. Installable artifactsPublished artifacts or installers with checksums where appropriate.
  4. Verified pathA Docker-first installation path tested from a clean environment for the stated platform.
  5. Acceptance recordA clear account of what was validated, by whom, and what remains unaccepted.

Why the language is strict A polished site, a local demo, or passing static checks can support a design claim. None of them, alone, make software publicly available or accepted.
